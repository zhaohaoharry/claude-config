Standing practice the user wants in EVERY research project (2026-06-18, his long-time Stata habit of a master do-list / run-order file, now generalized to Python and all languages). **Priority rule, all projects.**

Keep a **`PIPELINE.md`** at the project root: the programs listed in build order from raw data to final results, and under each program its **input files** and **output files**, so the chain `data/raw → data/clean → results → latex/tables|figures` is explicit and the programs are the bridges. This is the empirical-economics master-do-file convention.

**Crucial twist (from the hand-built-table lesson, see [[feedback_every_number_has_provenance]]):** the manifest is **GENERATED by a script**, not hand-maintained — a hand-kept run-order list drifts from the code exactly the way a hand-typed result does. The generator extracts each program's reads/writes directly from the source, so the map cannot silently disagree with the code.

**Reference implementation (Water and Crop):** `program/build_pipeline.py` scans all `.py`/`.do` for read/write calls (with path-variable resolution and f-string-suffix cleaning), anchors on the manuscript's `\input`/`\includegraphics` artifacts, traces dependencies backward to raw data, topologically orders, and writes `PIPELINE.md`. Companion `program/audit_provenance.py` flags any artifact with no generator. Run both before any `/sync-to-github` or submission.

**How to apply:** when starting or touching a project that lacks `PIPELINE.md`, copy and adapt `build_pipeline.py` + `audit_provenance.py`. Re-run after adding/changing any pipeline program. Treat `PIPELINE.md` as read-only output. Known v1 limitations of the extractor: approximate topological order, and dynamically-built/multi-line paths can be missed (e.g. `nass_{year}.csv` collapses to `nass_.csv`) — curate those by hand or improve the extractor, but never hand-author the whole file. Also recorded as a global Universal Rule in `~/.claude/CLAUDE.md`. See [[project_table2_provenance_rebuild]].
